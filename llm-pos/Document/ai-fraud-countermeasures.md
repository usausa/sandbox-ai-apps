# 小売不正対策へのAI活用 実現可能性マップ

現場で発生している実際の不正事例を、**LLM / Azure AI Foundry / Azure の各サービス**で対応できるかという観点で棚卸しし、
それぞれの手法を **実用性・実装難易度・環境導入難易度・リアルタイム処理可否・コスト** の5軸で評価したドキュメントです。
起点は既に作製済みの `llm-pos`（PosChecker）で、本書はその延長として「次に何を作れば現場の不正に効くか」を整理します。

> 本書はサービス選定・優先順位づけのための検討資料です。各手法の精度・コストは PoC で実測して確定してください。
> **同じ課題に対する対策でも、リアルタイム版とバッチ集計版で性質（実用性・難易度・コスト）が大きく異なる場合は、別項目に分けています。**

---

## 0. 前提：PosChecker で既にできていること

`PosChecker` は **POS トランザクション（テキスト/CSV）** から、決定論的な機械集計と LLM（Foundry / gpt-5.4-mini）の
ハイブリッドで、次の **5事象** を「店舗×担当者」主軸で検知します（詳細は `design.md`）。**これは現状すべて「バッチ集計版」**です。

| # | 既存の検知事象 | 今回の現場事例との対応 |
|---|---|---|
| 1 | ポイント不正付与（`PointAbuse`） | 「ポイントカード不正使用（同一カード連続スキャン・無断使用）」に直結 |
| 2 | かご抜け（`CartBypass`） | 「従業員と客の共謀・スキャン漏れ」に直結（POSログの範囲内で） |
| 3 | クーポン不正利用（`CouponAbuse`） | 「クーポン反復利用」に対応（偽造券の真贋判定は範囲外） |
| 4 | フリー返品不正（`ReturnFraud`） | 「返金操作の悪用」に直結 |
| 5 | 打ち直し不正（`RekeyFraud`） | 「打ち直し操作の悪用」に直結 |

**重要な気づき:** 現場で挙がった事例の多くは、すでに PosChecker のアーキテクチャ（決定論的集計＋LLM 判定）が
そのまま、あるいは小さな拡張で効く領域です。一方で **現金・映像・在庫** はテキスト外のデータが必要で、
別系統のサービス（画像/映像/データ基盤）を足す必要があります。

## 1. 評価軸の定義

各手法を **5軸**で評価します。★は軸ごとに向きが違うので注意してください。

- **実用性**（実運用での効果・ROI）: ★が多いほど良い。**★★★★★ = 効果が大きく投資対効果が高い** … ★☆☆☆☆ = 限定的
- **実装難易度**（＝AI/ソフトウェアの**ロジックを作る**難しさ）: ★が多いほど難しい。アルゴリズム・プロンプト・誤検知抑制の作り込み量。
- **環境導入難易度**（＝**現場環境へ入れる**難しさ。POS/機器連携・リアルタイム通信・カメラ等ハードの要否）: ★が多いほど難しい。
  - **★☆☆☆☆ 容易** … 既存のデータ/ファイル（POSログ等）で完結。**バッチはここ**。新たな機器・通信が不要。
  - **★★★☆☆ 中** … 撮影機器の追加、発行DB等への接続、または「締め」など離散イベントの連携が必要。
  - **★★★★☆ 難** … **リアルタイム通信**（POS/精算機イベントの常時連携）が必要。既存POSは即時イベント出力を持たないことが多く、改修・連携が重い。
  - **★★★★★ 最難** … **カメラ・センサ・精算機/ドロワー等のハード設置・改修**が必要。設置工事・回線・常時推論基盤を伴い、コストも跳ね上がる。
- **リアルタイム処理可否**:
  - **◎ リアルタイム** … 取引/イベント単位で即時（秒オーダー）。**発生時点で判定・ブロック・通知する価値がある**。
  - **○ 準リアルタイム** … 数分〜「締め直後」。ストリーミング/マイクロバッチ（Fabric eventstream・Azure Stream Analytics 等）。
  - **△ バッチ** … 日次/定期。**蓄積されたパターンが判定に必要**なため即時化の価値が薄い、または即時化が技術的に困難。
- **コスト**（処理コストの相対水準）: **低（¥）**=テキスト/LLM・OCR中心で安価 ／ **中（¥¥）**=画像の都度処理・帳票カスタム抽出・小中規模データ基盤 ／ **高（¥¥¥）**=映像の常時推論・大規模データ統合基盤の固定費・設備投資。

> **実装難易度と環境導入難易度は別物です。** 例えば「在庫の日次バッチ突合(B-4)」はAIロジック自体は平易（実装★寄り）でも、
> 複数システムのデータを繋ぐ環境整備が重い（環境★高め）。逆に「不審行動検知(D-2)」はモデルもハードも重い（両方高）。
> **多くの手法はAIの実装より、環境（通信・機器）の導入がボトルネック**になります。

**コストの目安（2026年時点の概算アンカー）**

| サービス | 概算単価 | 備考 |
|---|---|---|
| Foundry gpt-5.4-mini（テキストLLM） | 入力 $0.75 / 出力 $4.50（per 100万トークン） | `appsettings.json` 設定値。PosChecker 実測 **約 ¥1〜2/回**（週次バッチ） |
| Azure AI Vision Read（OCR） | **$1.5 / 1,000ページ** | 大量時は $0.6 まで逓減 |
| Document Intelligence（プリビルト/レイアウト） | $10 / 1,000ページ | 領収書・帳票レイアウト |
| Document Intelligence（カスタム抽出） | $30 / 1,000ページ | 手書き帳票の独自フォーマット。バーコード等アドオン +$6/1,000 |
| Azure AI Video Indexer | **尺課金（入力の分単位）** | 複数カメラの常時運用は分数が膨大になり**高額化**。要見積 |
| Microsoft Fabric / Synapse | 容量ベースの**固定費** | 処理量に依らず基盤を持つコストが発生 |

> **リアルタイムとコストの関係（重要な設計指針）:** リアルタイム処理は **LLM を毎取引呼ばない**のが原則。
> 取引時は**軽量な決定論ルール（ストリーミング）**で即時判定し、**LLM は「締め後のバッチ要約」「アラート発生時の深掘り」**に限定する。

| **D-4** | 万引き／内部不正の**判別** | ○〜△ | ★★★☆☆ | ★★★★★ | ★★★★★ | 高 | カメラ×POS×ドロワー相関 |
| **E-1** | セミセルフ**自動回収・返却機能** | ◎(ハード) | ★☆☆☆☆ | ★★★★☆ | ★★★★★ | 高 | 精算機ハード/システム改修主体 |

**環境難度の読み方:** **★1=バッチ（既存データで完結・最も入れやすい）**、**★4=リアルタイム通信（POS/精算機の常時イベント連携が必要）**、**★5=カメラ/センサ/ハード設置（工事・回線・常時推論でコストも最大）**。
コスト列と見比べると、**環境難度★5の手法（D群・E-1・B-3）はコストも「高」**に揃うことがわかります。

**着手順の目安:** まず環境難度★1の **A-2・A-4・B-7**、次に★2の **C-1・B-2** → ★3〜4の通信連携 **A-1・A-3・B-1・B-5**・在庫 **B-4** → 環境難度★5の **映像D群・在庫ストリーミングB-3・E-1** は最後に PoC 評価。
**原則として「同一課題はバッチ版（環境★低）で安く立ち上げ → 効果を確認してからリアルタイム/機器版（環境★高）へ投資」**を推奨します。

## 3. カテゴリ別の詳細

各項目の見出しに **RT / 実用 / 実装 / 環境 / コスト** を併記します。

### カテゴリ A：POSトランザクション分析（LLM＋Foundry の本領／PosChecker の延長）

#### A-1. POS不正の取引時リアルタイム検知（即時ルール）　RT ◎／実用 ★★★★☆／実装 ★★★☆☆／環境 ★★★★☆／コスト 低
- **対象:** ポイント連続スキャン、レシートなし返金、短時間の同一会員連続会計など、**単一取引/短時間窓で判定できる**シグナル。
- **やること:** POS イベントをストリームで受け、取引直後に検知して通知/要確認（LLM は呼ばない軽量ルール）。
- **環境難易度が高い理由:** ★4。**既存POSはリアルタイムのイベント出力を持たないことが多く**、取引ストリームを外部へ流す改修・連携が重い。AIロジック自体は軽い。
- **限界:** 「他担当者と比べた比率の偏り」など蓄積が要るシグナルは即時判定不可（残りは A-2 が担う）。

#### A-2. POS不正のバッチ・パターンスコアリング（現 PosChecker）　RT △／実用 ★★★★★／実装 ★★☆☆☆／環境 ★☆☆☆☆／コスト 低
- **対象:** ポイント不正付与、返品悪用、打ち直し悪用、ポイントカード偏り（蓄積パターン）。
- **やること:** 日次バッチで全店舗×全担当者をスコアリングし、閾値超過をアラート（現 PosChecker の実運用化）。
- **環境難易度が低い理由:** ★1。**既存のPOSログ/CSVエクスポートで完結**。新たな機器・通信が不要で、最も入れやすい。
- **実用性が最高位の理由:** 現場事例の中核を既存実装でカバー。発覚を「2〜3週間〜四半期」から日次へ短縮。

#### A-3. クーポン使用済み照合（スキャン時リアルタイム）　RT ◎／実用 ★★★★☆／実装 ★★★☆☆／環境 ★★★★☆／コスト 低
- **やること:** クーポンスキャン時に **使用済みリスト・有効期限・配信期間**へ即時照合し、再利用/期限切れをその場で弾く。
- **環境難易度が高い理由:** ★4。**POS/レジ側でスキャン時に照合APIを叩く改修**と、低遅延のクーポン履歴DBが要る。
- **A-4 との違い:** こちらは「使う瞬間の照合（防止）」、A-4 は「事後のパターン検知（発見）」。

#### A-4. クーポン反復利用のバッチ検知（現 CouponAbuse）　RT △／実用 ★★★★☆／実装 ★★☆☆☆／環境 ★☆☆☆☆／コスト 低
- **やること:** 既存 `RepeatedCouponByMember`。同一会員の反復、配信期間との乖離を期間横断で検出。
- **環境難易度が低い理由:** ★1。既存の販促データで完結。**A-3 で水際を固め、A-4 で取りこぼしを発見**する二段構え。
- **限界:** 物理的な偽造券の真贋は対象外（→ C-3）。

#### A-5. かご抜け・共謀のバッチ検知（現 CartBypass）　RT △／実用 ★★★☆☆／実装 ★★★☆☆／環境 ★☆☆☆☆／コスト 低
- **やること:** 既存 `CartBypass`（同一JAN/用途の複数購入の頻発・特定会員への偏り）。
- **環境難易度が低い理由:** ★1（POSのみ）。ただし**確証**を得るには重量センサ/映像が必要で、その場合は環境★が一気に★5へ跳ねる（→ D-4）。
- **限界:** POSログには「スキャンしなかった商品」が残らず、間接検知に留まる。

### カテゴリ B：データ統合・異常検知（Foundry ＋ データ基盤）

数値・イベントを時系列／突合で見る領域。**Azure Anomaly Detector は2026/10廃止**のため、異常検知は **Microsoft Fabric**
（`microsoft/anomaly-detector` を統合）や Azure ML を使い、**説明と要約を Foundry の LLM** に任せる構成を推奨。

#### B-1. 現金差異の締め直後アラート（準リアルタイム）　RT ○／実用 ★★★★☆／実装 ★★★☆☆／環境 ★★★☆☆／コスト 低〜中
- **やること:** レジ締めデータが入った瞬間に「理論在高 − 実在高」差異を計算し、閾値超でその場で通知。
- **環境難易度が中位の理由:** ★3。**「締め」という離散イベント**単位の連携でよく、取引の常時ストリーミング（A-1の★4）ほど重くない。ただし現金点検データのデジタル取得（C-1）が前提。
- **実用性が高い理由:** 「手入力で誤魔化し可能・発覚が1ヶ月遅れ」という現状の核心を解消。

#### B-2. 現金差異の時系列パターン分析（バッチ）　RT △／実用 ★★★★☆／実装 ★★★☆☆／環境 ★★☆☆☆／コスト 低
- **やること:** 差異・ドロワー開閉・NoSale・訂正の時系列を担当者/曜日/時間帯で横断分析し、慢性的な偏りを検出。Foundry が要約。
- **環境難易度が低めの理由:** ★2。バッチ集計だが、入力に現金点検データのデジタル化（C-1）が必要な分だけ A-2 より上。
- **B-1 との違い:** B-1 は「その日の異常を即通知」、B-2 は「期間の傾向から黒に近い人物を炙り出す」。

#### B-3. 在庫差異のストリーミング早期警告（準リアルタイム）　RT ○／実用 ★★★★★／実装 ★★★★★／環境 ★★★★★／コスト 高
- **やること:** 入荷・売上・返品・廃棄イベントをストリームで突合し、理論在庫を逐次更新。閾値超を発生直後に警告。
- **環境難易度が最高位の理由:** ★5。**複数システムを常時ストリーミングで連携**する必要があり、最も重い。まずは下の B-4 で十分なことが多い。

#### B-4. 在庫差異の日次バッチ突合（バッチ）　RT △／実用 ★★★★★／実装 ★★★☆☆／環境 ★★★★☆／コスト 中
- **やること:** 入荷×POS売上×返品/廃棄を夜間バッチで突合し、理論在庫を日次更新。Foundry が原因仮説を提示。
- **環境難易度が高めの理由:** ★4。バッチでも**複数システムのデータ連携**が要る（現状「各システム間の連携不足」が最大の障壁）。ただし常時ストリーミング不要な分 B-3 より軽い。
- **推奨:** **在庫はまず B-4（日次）で立ち上げ、必要なら B-3（ストリーミング）へ拡張**するのが費用対効果良。

#### B-5. 取り忘れ現金・未回収の即時アラート（準リアルタイム）　RT ○／実用 ★★★☆☆／実装 ★★☆☆☆／環境 ★★★★☆／コスト 低
- **やること:** 精算機の「取り忘れ/未回収」イベント発生後、一定時間内に回収記録が立たなければ即アラート。
- **環境難易度が高い理由:** ★4。**精算機が取り忘れ/未回収イベントを外部出力できるか**という機器連携が前提。AIロジックは軽い。
- **注意:** 防止（自動回収・返却）はハード主体（→ E-1）。AIは「検知・抑止」側。

#### B-6. 着服の事後突合・偏り分析（バッチ）　RT △／実用 ★★★☆☆／実装 ★★☆☆☆／環境 ★★★☆☆／コスト 低
- **やること:** 取り忘れ発生 vs 回収/入庫記録を期間で突合し、特定従業員への偏り・回収漏れ率を分析。
- **環境難易度が中位の理由:** ★3。バッチだが精算機ログ＋回収記録（紙なら要デジタル化）の取得が要る。B-5（即時）と役割分担。

#### B-7. レジ処理速度・スキャン時間の個人別分析（バッチ）　RT △／実用 ★★★☆☆／実装 ★★☆☆☆／環境 ★☆☆☆☆／コスト 低
- **やること:** 担当者別処理速度・商品別スキャン時間・訂正率を集計（効率化＋不正の先行指標）。Foundry でレポート要約。
- **環境難易度が低い理由:** ★1。タイムスタンプは既存 POS ログにあり、追加機器・通信が不要。

### カテゴリ C：画像認識（マルチモーダル LLM / Document Intelligence / Vision）

#### C-1. 手書きレジ点検票・金種シートのデジタル化　RT ○／実用 ★★★★☆／実装 ★★☆☆☆／環境 ★★☆☆☆／コスト 低
- **やること:** 点検票/金種シート（手書き帳票）を撮影 → OCR で金種別枚数・合計を構造化 → B-1/B-2 の入力へ。
- **環境難易度が低めの理由:** ★2。**既存のタブレット/スマホのカメラ**で撮影でき、専用設置や常時通信は不要。
- **実用性が高い理由:** 手入力の誤魔化しを排し、現金監視（B-1/B-2）の入力を自動生成。地味だが確実で安い。

#### C-2. 現金写真からの自動金種計算　RT ◎／実用 ★★★☆☆／実装 ★★★☆☆／環境 ★★★☆☆／コスト 中
- **やること:** ドロワー/現金の写真から金種・枚数を推定し合計を算出。
- **環境難易度が中位の理由:** ★3。撮影端末に加え、**精度確保のための撮影スタンド/プロトコル標準化**が要る。
- **実用性の注意:** 重なり・遮蔽で精度が落ち、釣銭1枚単位の正確さは写真だけでは担保しづらい。本命は計数機/つり銭機からの計数値で、AIは補助。C-1 の方が確実。

#### C-3. 偽造クーポン・ギフト券の真贋補助　RT ◎／実用 ★★★☆☆／実装 ★★★☆☆／環境 ★★★☆☆／コスト 低〜中
- **やること:** 券面を撮影 → OCR で券種/番号/有効期限/発行元を抽出 → 発行DB・使用済みリスト・有効期間と即時照合。
- **環境難易度が中位の理由:** ★3。レジでの撮影端末＋**発行/使用済みデータへの接続**が要る（受理の瞬間に照合するため）。
- **限界:** 番号・期限・発行元の論理的真贋は強力に検知。精巧な物理偽造（紙質・ホログラム）は画像のみでは限界で、最終判断は人＋台帳。

### カテゴリ D：映像解析（カメラ＋ Azure AI Video Indexer）

**注意: 旧 Azure Vision Spatial Analysis（人数・行列カウント）は2025/3/30に廃止済み**。後継は Azure AI Video Indexer、
静止画フレームへの物体検出（Image Analysis 4.0 / Custom Vision）、または専用のリテール映像解析。**この群は環境導入★5（カメラ設置・回線・常時推論）でコストも最大**。

#### D-1. レジ待ち人数検知・応援要請　RT ◎／実用 ★★★☆☆／実装 ★★★★☆／環境 ★★★★★／コスト 高
- **やること:** レジ前カメラのフレームから待ち人数/滞留を推定し、閾値超で応援通知。
- **環境難易度が最高位の理由:** ★5。**カメラ設置・回線・推論基盤**が必須。常時尺課金でコストも高。
- **実用性が中位の理由:** 接客改善には有効だが不正抑止効果は間接的。

#### D-2. 不審行動の映像検知　RT ◎／実用 ★★☆☆☆／実装 ★★★★★／環境 ★★★★★／コスト 高
- **実装/環境/コスト:** いずれも最大級。汎用「不審行動」検知は**誤検知が多く運用破綻しやすい**。POS連動の特定イベント（NoSale時に手元が映る等）に絞る方が現実的。

#### D-3. 顧客動線分析　RT △／実用 ★★☆☆☆／実装 ★★★★☆／環境 ★★★★★／コスト 中〜高
- **位置づけ:** 不正対策というより**売場最適化**。カメラ/センサ基盤が要り、不正対策の優先度は低い。

#### D-4. 万引き／内部不正の判別　RT ○〜△／実用 ★★★☆☆／実装 ★★★★★／環境 ★★★★★／コスト 高
- **やること:** 在庫差異（B-3/B-4）の発生時点に **POSイベント・ドロワー開閉・カメラ映像を時刻で相関**させ、Foundry が「内部不正/万引き/誤登録」を根拠付きで提示。
- **環境難易度が最高位の理由:** ★5。**カメラ＋POS＋ドロワーの3系統を時刻同期で統合**する必要がある。まずは B-2＋B-4（データのみ）で大半の切り分けは可能。

## 4. 横断アーキテクチャの提案（リアルタイム経路とバッチ経路）

**コスト最適化の要点 = リアルタイム経路には LLM を置かない。** 取引/イベント時は軽量ルールで即時判定し、
LLM（Foundry）は「締め後のバッチ要約・スコアリング」「アラート発生時の深掘り」だけに使う。

```text
                 ┌──────── リアルタイム経路（◎/○・低コスト・LLMなし／環境★3〜4: 常時通信が要る） ────────┐
 [POS/精算機/在庫] →  Stream Analytics / Fabric eventstream  →  軽量決定論ルール  →  即時アラート/水際ブロック
                 └  A-1, A-3, B-1, B-5, B-3, (C-3は受理時)                                          ┘

                 ┌──────── バッチ経路（△・低〜中コスト・LLMで判定/要約／環境★1〜4: 既存データ連携） ────┐
 [POS/現金/在庫/帳票] → 決定論的な集計・突合（説明できる特徴量） → Azure AI Foundry(gpt/マルチモーダル) → スコア・理由・推奨
   ・帳票は Document Intelligence でデジタル化（C-1）                A-2,A-4,A-5,B-2,B-4,B-6,B-7,D-4
                 └  → 店舗リスクダッシュボード / 日次レポート                                         ┘

 [カメラ] → Video Indexer / 物体検出（D-1,D-2,D-3）… 環境★5・高コスト・別系統。PoCで費用対効果を判定。
```

- **共通パターン:** 生データを直接 LLM に投げず、**決定論的に集計・突合して「説明できる特徴量」を作り**、LLM には判定と要約をさせる（誤検知抑制・トークン削減・説明性。PosChecker と同じ思想）。
- **束ね役に Foundry Agent Service:** 各検知結果を店舗単位の1レポートへ統合。Azure AI Search（規程/過去事例）や Fabric データを tool として参照可能。
- **廃止サービスの回避（重要）:** **Spatial Analysis（2025/3廃止）・Anomaly Detector（2026/10廃止）は新規採用しない**。人数/行列は Video Indexer 等、時系列異常検知は Microsoft Fabric を使う。

フェーズが進むほど環境導入難易度とコストが上がる構成。

| フェーズ | 内容 | 含む手法 | 環境難度 | ねらい |
|---|---|---|:---:|---|

## 6. 共通の注意点

- **環境（通信・機器）の導入が最大のボトルネック:** AI実装より、**POS/精算機のリアルタイム連携・カメラ設置**の方が重い。「必要な機器・通信が既にあるか」を先に確認する。
- **リアルタイム経路に LLM を多用しない:** 毎取引の LLM 呼び出しはレイテンシ・コスト増。即時判定は決定論ルール、LLM はバッチ/深掘りに。
- **誤検知と説明性:** スコアだけでなく**根拠（特徴量・該当キー）を必ず提示**。最終判断は人。
- **データ取得が律速:** AI 能力より**ログ・帳票・映像・精算機イベントが取れているか**が成否を決める。
- **プライバシー／コンプライアンス:** 映像・従業員監視は個人情報保護法と労務配慮が必須。利用目的明示・アクセス制御・保管期間の規程整備を前提に。
- **コスト感:** テキスト/帳票/OCR は安価（LLM 約¥1〜2/回、OCR $1.5/1,000）。**映像の常時推論・ストリーミング基盤の固定費は桁違い**。
- **廃止サービス:** Spatial Analysis・Anomaly Detector は採用しない（前述）。
