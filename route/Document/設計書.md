## 1. 概要・目的

電気設備点検の調査員が、その日に訪問すべき調査先の一覧をもとに、**1 日・調査員 1 名単位**で巡回する順序（**足順**）を決定するアプリケーション。

- 移動時間と各訪問先の指定時間帯を考慮した足順を**アルゴリズムで算出**する。
- 算出した足順を **Microsoft Foundry（LLM）に渡してルール検証と改善提案**を受け取る（ハイブリッド構成）。
- 結果として、時系列の訪問順・到着/出発予定・移動量・違反/リスク・改善提案を画面に表示する。

ねらいは「最適化アルゴリズム」と「LLM によるルール検証・助言」を組み合わせ、機械的に最適化しきれない業務知識・常識（時間帯厳守、残業回避、天候配慮など）を AI が補完する構成をサンプルとして示すこと。

## 2. スコープと前提

### 対象（In Scope）
- 調査員 1 名・1 日分の足順決定。
- 訪問先一覧（CSV）と共通設定をインプットとする。
- 単一の出発拠点（事業所）から出発し、原則として事業所へ帰着する単一ルート。
- 移動手段は基本「車」。徒歩・自転車・公共交通も平均速度の違いとして扱える。

### 対象外（Out of Scope, 将来拡張）
- 複数調査員への割当（配車最適化）、複数日にわたる計画。
- 実地図 API（経路探索・ジオコーディング・リアルタイム交通）連携。本サンプルは**距離計算を地図 API に依存せず**、訪問先の緯度経度を直接インプットし、直線距離（Haversine）＋道路迂回係数で移動時間を近似する（足順の地図表示には Leaflet＋OpenStreetMap タイルを用いる）。
- 実績の記録・打刻・GPS 照合。

### 前提
- 各訪問先には緯度経度が付与されている（住所のジオコーディングは将来拡張）。サンプルは山梨県甲府市の地点で、出発/帰着拠点は甲府市役所を既定とする。
- 時刻はすべて当日内（日跨ぎなし）で扱う。
- 足順の地図表示は OpenStreetMap タイル（Leaflet）を使用する。

## 3. 想定ユーザーとユースケース

| ユーザー | 主なユースケース |
|---|---|
| 点検配車担当 / 班長 | 朝、当日の訪問先一覧を取り込み、調査員ごとの足順を作成・確認し、無理がないか AI レビューで点検する。 |
| 調査員本人 | 自身の当日分一覧から足順案を作り、指定時間帯や残業の問題がないかを確認する。 |

基本シナリオ:
1. 当日の訪問先一覧 CSV を用意（または同梱サンプルを利用）。
2. 画面にアップロード。
3. 足順案・到着予定・移動量・AI レビュー（違反/改善提案）を確認。
4. 提案を踏まえて手動で調整、または入力を直して再実行。

## 4. 用語定義

| 用語 | 定義 |
|---|---|
| 足順 | 1 日の訪問先を回る順序。出発（事業所）→訪問→…→昼休憩→…→帰着（事業所）の時系列。 |
| 立ち寄り（Stop） | 足順上の 1 ステップ。出発拠点 / 訪問先 / 昼休憩のいずれか。 |
| 指定時間帯（Time Window） | 訪問先側で指定された訪問可能/希望の時間帯（例: 午前中、14:00–16:00）。 |
| 厳守 / 希望 | 指定時間帯の制約強度。厳守＝ハード制約、希望＝ソフト制約。 |
| ハード制約 | 必ず満たすべき制約（勤務時間、厳守時間帯など）。違反する解は採用しない。 |
| ソフト制約 | できれば満たしたい制約（希望時間帯など）。違反は許容しつつ評価・是正対象とする。 |
| 実現性スコア | AI が返す足順全体の品質・実行可能性（0–100、高いほど良い）。 |

## 5. システム全体構成

ハイブリッド構成（決定的アルゴリズム＋ LLM）。

```text
[入力] 訪問先一覧CSV ＋ 共通設定（既定値）
        │
        ▼
  VisitCsvLoader … CSVを VisitTarget へ変換・検証
        │
        ▼
  TravelTimeMatrixBuilder … 2地点間の移動距離/時間を推定（Haversine×迂回係数÷速度×天候係数）
        │
        ▼
  RouteOptimizer … 時間帯を考慮した最近傍法で足順を算出（昼休憩挿入・帰着・未割当判定）
        │            └─ 出力: RoutePlan（順序・到着/出発・移動量・残業・違反・未割当）
        ▼
  RouteReviewAnalyzer … RoutePlan を Foundry(LLM) に渡し、ルール検証＋改善提案を取得
        │            └─ 出力: RouteReviewResult（実現性スコア・違反・改善提案・注意メモ）
        ▼
  RoutePlannerService … 上記を統括し RoutePlanResult を生成
        │
        ▼
[表示] Blazor 画面（足順テーブル・サマリー・違反/改善・プレビュー）
```

役割分担:
- **アルゴリズム**: 移動時間最小化と時間帯/勤務時間という定量制約の充足（再現性・高速）。
- **AI**: 定量化しにくい業務ルール・常識（厳守超過の指摘、残業/未割当の是正、天候配慮、並べ替え提案）の検証と助言。

## 6. 処理フロー

| # | フェーズ | 入力 | 処理 | 出力 |
|---|---|---|---|---|
| 1 | 取込 | CSV ストリーム | ヘッダ解析、行を `VisitTarget` 化、必須項目検証 | `IReadOnlyList<VisitTarget>` |
| 2 | 距離/時間 | 2 地点座標＋共通設定 | Haversine 距離→道路迂回→速度/天候換算 | `TravelLeg`（km・分） |
| 3 | 最適化 | 訪問先群＋共通設定 | 時間帯考慮の最近傍選択、昼休憩挿入、帰着、未割当判定 | `RoutePlan` |
| 4 | AI 検証 | `RoutePlan`＋共通設定 | 特徴を JSON 化→LLM→違反/改善を JSON 取得→正規化 | `RouteReviewResult` |
| 5 | 統括 | 上記 | 結果束ね・プレビュー抽出 | `RoutePlanResult` |
| 6 | 表示 | `RoutePlanResult` | 画面描画 | UI |

## 7. インプット項目設計（中核）

インプットは 2 系統。**A. 共通インプット（1 日・調査員 1 名単位）** と **B. 調査先（訪問先）ごとのインプット**。

各項目には「**制約区分**」を付し、アルゴリズム/AI でどう使うかを明示する。

- 制約区分の凡例
  - **ハード**: 必ず満たす（違反解は不採用）
  - **ソフト**: できれば満たす（違反は許容しつつ是正対象）
  - **係数**: 移動/所要時間の補正に使う数値
  - **既定補完**: 訪問先側で未指定のときの既定値を与える
  - **メタ**: 表示・連絡・安全のための情報（最適化の数式には直接入らない）

### 7.A 共通インプット（`CommonSettings`）

| 論理項目 | プロパティ | 型 | 必須 | 既定値 | 制約区分 | 設計意図・利用箇所 |
|---|---|---|---|---|---|---|
| 業務日 | `WorkDate` | date | ○ | 当日 | メタ | 対象日。表示・AI への文脈。 |
| 調査員 ID | `InspectorId` | string | ○ | INS-01 | メタ | 1 名単位の識別。 |
| 事業所住所 | `OfficeAddress` | string | ○ | （丸の内例） | メタ | 出発/帰着拠点の表示。 |
| 事業所座標 | `OfficeLatitude` / `OfficeLongitude` | double | ○ | 35.681 / 139.767 | ハード | 足順の始点・終点。移動時間行列の基準。 |
| 事業所へ帰着 | `ReturnToOffice` | bool | △ | true | ハード | 最終立ち寄りを事業所に固定するか。 |
| 勤務開始/終了 | `WorkStart` / `WorkEnd` | time | ○ | 09:00 / 18:00 | ハード | 1 日の稼働枠。終了超過は残業。 |
| 出発時刻 | `DepartTime` | time? | △ | 未指定=勤務開始 | ハード | 拠点出発時刻。 |
| 昼休憩 時間帯 | `LunchStart` / `LunchEnd` | time | △ | 12:00 / 13:00 | ソフト | 休憩枠の目安。 |
| 昼休憩 所要 | `LunchMinutes` | int | △ | 60 | ハード | 休憩で消費する時間。0 で休憩なし。 |
| 移動手段 | `TravelMode` | enum | ○ | Car | メタ/係数 | 既定速度の前提。表示と速度設定の根拠。 |
| 平均移動速度 | `AverageSpeedKmh` | double | △ | 25（市街地・車） | 係数 | 距離→移動時間換算。手段により変える。 |
| 道路迂回係数 | `RoadDetourFactor` | double | △ | 1.3 | 係数 | 直線距離→実走行距離の補正。 |
| 天気予報 | `Weather` | enum | △ | Sunny | メタ/係数 | 移動余裕・安全配慮。AI の注意喚起材料。 |
| 天候 所要延長係数 | `WeatherDurationFactor` | double | △ | 1.0 | 係数 | 雨/雪/猛暑時の移動時間延長（例: 雨 1.15）。 |
| 種別別 平均調査時間 | `GeneralServiceMinutes` / `BusinessServiceMinutes` | int | ○ | 20 / 40 | 既定補完 | 訪問先で `ServiceMinutes` 未指定時の所要。 |
| 訪問間バッファ | `BufferMinutes` | int | △ | 2 | 係数 | 次の訪問先への徒歩移動・メーター確認等の予備時間（戸別点検の徒歩巡回を想定）。同一建物内の連続調査では 0 とする。 |
| 1 日の最大訪問件数 | `MaxVisits` | int? | △ | 無制限 | ハード | キャパ上限。 |
| 残業可否 | `AllowOvertime` | bool | △ | false | ハード | 勤務終了超過を許容するか。 |
| 最大残業 | `MaxOvertimeMinutes` | int | △ | 0 | ハード | 許容する超過分（`AllowOvertime` 時）。 |
| 指定時間帯 既定厳守度 | `DefaultWindowStrictness` | enum | △ | Preferred | ソフト | 訪問先で未指定時の既定。 |
| 最適化方針 | `Objective` | enum | △ | MinimizeTravel | メタ | 目的関数の重み付け方針（移動最小/指定厳守優先）。 |

> 補足: 「平均調査時間」はご指摘どおり重要要素のため、**種別ごとの既定値（共通設定）** と **訪問先ごとの個別上書き（`ServiceMinutes`）** の二段構えとした。これにより、ほとんどの訪問先は個別入力なしで妥当な所要時間が割り当てられ、特殊な訪問先のみ個別指定すれば足りる。

### 7.B 調査先（訪問先）ごとのインプット（`VisitTarget`）— CSV 1 行 = 1 訪問先

| 論理項目 | 列名 | 型 | 必須 | 例 | 制約区分 | 設計意図・利用箇所 |
|---|---|---|---|---|---|---|
| 訪問先 ID | `VisitId` | string | ○ | V001 | メタ | 一意キー。違反/提案の参照先。 |
| 顧客名 | `CustomerName` | string | △ | 佐藤邸 / ○○商店 | メタ | 表示用。 |
| 住所 | `Address` | string | ○ | 東京都… | メタ | 確認・表示用。 |
| 緯度 / 経度 | `Latitude` / `Longitude` | double | ○ | 35.70 / 139.77 | ハード | 移動時間行列（ジオコーディング代替）。 |
| 種別 | `Category` | enum | ○ | 一般 / 業務 | 既定補完/メタ | 既定調査時間・優先度傾向。 |
| 調査所要時間 | `ServiceMinutes` | int? | △ | 30 | ハード | 個別の所要（未指定は種別既定で補完）。 |
| 指定時間帯 開始/終了 | `WindowStart` / `WindowEnd` | time? | △ | 14:00 / 16:00 | ハード/ソフト | 訪問可能/希望の枠。厳守度で強度が変わる。 |

| アクセス制約 | `AccessNote` | string? | △ | 駐車不可 / 立会要 / 鍵 | メタ | 注意喚起・AI のリスク指摘材料。 |

| 連絡先 | `ContactPhone` | string? | △ | 03-… | メタ | 連絡用。 |

### 7.C 指定時間帯の表現

- 自由記述の「午前中」「14:00-16:00」などは、取り込み時に **開始/終了の `time`（HH:mm）に正規化**することを基本とする。
  - 例: 午前中 → 09:00–12:00、午後 → 13:00–17:00。
- 厳守区分（`WindowStrict`）で制約強度を表す。**厳守**は最適化で必達（守れない候補は採用しない）、**希望**は超過を許容しつつ違反として記録・AI 是正対象。
- 片側のみ（開始のみ/終了のみ）の指定も許容する。

### 7.D バリデーション方針

- 必須: `VisitId`、`Latitude`/`Longitude`（数値）。欠落・不正は取込時にエラー。
- `VisitId` が空の行はスキップ。データ 0 件はエラー。
- enum・時刻・真偽は日本語/英語の表記ゆれを許容してパース（例: 業務/Business、厳守/Strict、高/High、true/要/○）。不正値は安全側の既定にフォールバック。

## 8. データ形式

### 8.1 訪問先 CSV（UTF-8・ヘッダ付き）

列順（推奨ヘッダ）:

```csv
VisitId,CustomerName,Address,Latitude,Longitude,Category,ServiceMinutes,WindowStart,WindowEnd,WindowStrict,Priority,AppointmentRequired,WorkType,BuildingGroupId,AccessNote,HazardNote,ContactPhone
V003,中央ビル管理室,山梨県甲府市中央4-5-1,35.6620,138.5720,業務,40,,,,高,true,故障,B-CHUO,駐車場は地下P,,055-100-0003
V005,国母商会,山梨県甲府市国母5-1-1,35.6380,138.5450,業務,30,13:00,15:00,希望,中,false,交換,,立会い必要,,055-100-0005
```

- 任意列は空欄可。`ServiceMinutes` 空欄は種別既定で補完。
- 時刻は `HH:mm`。

### 8.2 共通設定 JSON（サンプル）

本サンプルアプリでは共通設定は**既定値**で動作するが、将来のフォーム入力/取り込み用に形を定義する。`wwwroot/samples/common-settings.sample.json` を参照。

### 8.3 同梱サンプル（難易度別 3 種）

| ファイル | 内容 | ねらい |
|---|---|---|
| `kofu-dense-90.csv` | 狭域・戸建中心の約 90 件（一定地域の隣家を順に巡回、集合住宅は少数）、調査は短時間 | 戸別点検型の基本的な足順生成の確認 |
| `kofu-window-100.csv` | 狭域・戸建中心の約 100 件、午前中/14:00–16:00 等の厳守・希望が多い | 時間帯の取り合い・希望超過・未割当の検証 |
| `kofu-tight-120.csv` | 狭域に戸建を多数含む約 120 件、件数が処理能力を超える | 残業・未割当が出る境界ケース |

> サンプルはすべて山梨県甲府市**中心部の狭い範囲（約 1.3km 四方）**の地点。出発/帰着拠点は甲府市役所（35.6618, 138.5683）を既定とする。戸建を中心に近接した家が街区状に並び、隣家を順に巡回する想定。集合住宅は少数で、同一座標に複数住戸を持ち（同一 `BuildingGroupId`）、部屋番号は `CustomerName`、階は `AccessNote` で表す。

## 9. 出力設計

### 9.1 足順（`RoutePlan` / `RouteStop`）

- `RouteStop`: `Order`（順）, `Kind`（拠点/訪問/休憩）, `StopId`, `Label`, `Visit`（訪問先詳細）, `Arrival`/`Departure`, `ServiceMinutes`, `TravelKmFromPrev`/`TravelMinutesFromPrev`, `WaitMinutes`, `WindowViolation`, `Note`。
- `RoutePlan`: 立ち寄り列, `VisitCount`, 総移動距離/時間, 総調査/待ち時間, 開始/終了時刻, `OvertimeMinutes`, `WindowViolationCount`, `UnassignedVisits`。

### 9.2 AI レビュー（`RouteReviewResult`）

- `FeasibilityScore`（0–100）, `Summary`, `RecommendedAction`, `RuleViolations[]`, `ImprovementSuggestions[]`, `RiskNotes[]`。

### 9.3 画面表示項目

実現性スコア、足順サマリー（件数・総移動・稼働時間・残業・違反/未割当）、AI レビュー要約、**地図**、足順テーブル（時系列）、ルール違反テーブル、改善提案カード、注意メモ、未割当一覧、入力プレビュー。

### 9.4 地図表示

足順を地図上に可視化する。**Leaflet ＋ OpenStreetMap** タイルを用い、API キー不要で表示する。

- 各立ち寄り（拠点・訪問先）を**訪問順の番号付きマーカー**で表示（拠点＝青、時間枠超過＝赤、通常＝緑）。昼休憩は地点を持たないため地図には出さない。
- 立ち寄りを順に結ぶ**ポリライン**で巡回経路を示す（直線であり、実地図上の経路ではない）。
- 表示範囲はマーカー群に自動フィット。実装は `wwwroot/js/routemap.js` で、Blazor の JS 相互運用により `RoutePlanResult` から地点（順・ラベル・緯度経度・種別・違反フラグ）を渡す。

## 10. 足順最適化アルゴリズム設計

### 10.1 移動時間の推定（`TravelTimeMatrixBuilder`）

- 直線距離: Haversine（地球半径 6371km）。
- 実走行距離 = 直線距離 × `RoadDetourFactor`。
- 移動時間（分）= 実走行距離 ÷ `AverageSpeedKmh` × 60 × `WeatherDurationFactor`（切り上げ）。

### 10.2 足順生成（`RouteOptimizer`、時間帯考慮の最近傍法）

1. 出発拠点・出発時刻から開始。
2. ループ:
   - 昼休憩: 未取得かつ現在時刻が `LunchStart` 以降なら昼休憩を挿入。
   - 残りの訪問先について、移動（＋バッファ）後の到着時刻・着手時刻（窓開始まで待機）・所要・終了を計算。直前の立ち寄りと同一建物（同一 `BuildingGroupId`）の候補はバッファを 0 とし、集合住宅の住戸を連続して回れるようにする。
   - 同一建物・同一時間帯の住戸は**名称順（部屋番号順）**に回す。どの建物へ向かうかはスコアで決め、その建物・同一時間帯の中では最小名称の住戸へ寄せる（同一座標では優先度ペナルティ差で順序が乱れるため、名称順で固定。例: 201/101/301/102 → 101/102/201/301）。異なる時間帯の住戸は巻き込まない。
   - 地域結束: 現在地域（緯度経度を約 250m 四方のセルに丸めたキー）に**時間帯指定の無い**未訪問が残る間は、別地域へ移る時間帯指定なし候補に大きな擬似コストを加え、「隣接地域を 2 回に分けて回る」分割訪問を抑制する。ただし終了が間近（既定 10 分以内）に迫る時間帯指定訪問があるときは結束を停止し、時間帯遵守を優先する（空間的まとまりより時間帯を上位に置く）。
   - **厳守**かつ窓終了超過の候補は不採用。**希望**の超過は違反フラグ付きで採用可。
   - 終了が勤務終了（＋許容残業）を超える候補は不採用。
   - 選択コスト = 着手時刻 ＋ 0.1×移動分 ＋ **0.1×移動距離(km)** ＋ 優先度ペナルティ（高 −20 / 低 +10）＋ 窓なし +5。最小の候補を選択（早く着手でき、近く、優先度が高いものを優先）。
   - 移動「分」は切り上げのため近距離だと候補が同点になり入力順に落ちる（例: 地理的に 1→2→3 の一直線が 1→3→2 になる）。これを避けるため実距離(km)をごく小さな重みで加え、分が同点のときは地理的に近い候補を先に選ぶ（同一地域・同一時間帯内の並びを地理順に近づける）。1 分の差は覆さない。
   - 採用先へ移動。`MaxVisits` 到達で打ち切り。
3. 採用可能な候補が尽きたら、`ReturnToOffice` なら事業所へ帰着。
4. 残った訪問先は **未割当**。残業 = 帰着時刻 − 勤務終了（負なら 0）。

### 10.3 目的関数と制約

- ハード制約: 勤務時間枠、厳守時間帯、最大訪問件数、（許容内の）残業。
- ソフト制約: 希望時間帯（違反は許容・記録）、優先度。
- 最小化対象: 総移動時間（＋着手の早さ）。`Objective` により移動最小/指定厳守優先の重みを切替（将来拡張）。

### 10.4 既知の限界 / 将来拡張

- 現状は最近傍法ベースのヒューリスティック。「隣接地域の分割訪問」は地域結束（同一セルの時間帯指定なし訪問を先に片付ける）で緩和済みだが、時間帯指定が多い日は枠優先のため結束効果は限定的（空間的まとまりと時間帯遵守はトレードオフ）。**2-opt / Or-opt の局所探索**で移動時間をさらに短縮する余地がある（時間枠の実行可能性を保ちつつ入れ替え）。コード上に TODO を明示。
- 昼休憩は簡易（現在時刻が窓開始を越えた時点で取得）。窓中央での固定や前後の最適挿入は将来拡張。

## 11. AI ルール検証・改善設計

### 11.1 役割

算出済みの足順を入力に、**ルール違反の検証**と**具体的な改善提案**を行う（足順そのものは生成しない）。

### 11.2 入力ペイロード（LLM へ、camelCase JSON）

- `workDate` / `inspectorId` / `office`。
- `constraints`: 勤務時間、昼休憩、移動手段、平均速度、天候、残業可否、最適化方針。
- `planSummary`: 訪問件数、総移動距離/時間、総調査/待ち、開始/終了、残業、時間枠違反数、未割当数。
- `stops[]`: 各立ち寄りの順・種別・ID・ラベル・**位置情報（住所・緯度経度）・建物グループ**・カテゴリ・指定時間帯・厳守度・優先度・到着/出発・所要・移動分・待ち・違反フラグ。位置情報は地域の分割訪問（移動効率）を AI が検出するために渡す。
- `unassigned[]`: 未割当の訪問先（ID・種別・時間帯・厳守度・優先度）。

### 11.3 システムプロンプト方針（`Prompts/route_reviewer.txt`）

- 役割: 点検足順のレビュー担当。**JSON のみ**で回答。
- 検証観点: 厳守時間帯の超過、残業、未割当、昼休憩の確保、天候・アクセスのリスク、**移動効率（地域の分割訪問）**。
- 移動効率: 時間帯指定が無いのに隣接地域を 2 回に分けて回る分割訪問を、位置情報（緯度経度・住所）・建物グループから検出し、`detour` として指摘・`reorder` を提案。時間帯指定のための移動は問題視しない。
- スコア: `feasibility_score` は 0–100、**高いほど良い**（実現性・品質）。
- 提案: `reorder` / `move_earlier` / `move_later` / `call_ahead` / `split_day` 等を対象 ID 付きで。

### 11.4 出力契約（JSON）

```json
{
  "feasibility_score": 0,
  "summary": "<全体評価>",
  "recommended_action": "<最優先の対応>",
  "rule_violations": [
    {"stop_id": "V003", "type": "time_window|overtime|lunch|capacity|access|detour|other", "severity": "high|medium|low", "detail": "<内容>"}
  ],
  "improvement_suggestions": [
    {"action": "reorder|move_earlier|move_later|call_ahead|split_day|other", "target_stop_ids": ["V003"], "reason": "<理由>", "expected_effect": "<期待効果>"}
  ],
  "risk_notes": ["<天候・アクセス等の注意>"]
}
```

### 11.5 呼び出しと正規化

- 呼び出し: `Microsoft.Extensions.AI` の `IChatClient.GetResponseAsync`、`ChatOptions { Temperature = 0, ResponseFormat = ChatResponseFormat.Json }`（inspector と同一）。
- 応答解析: `PropertyNameCaseInsensitive = true`。失敗時は既定メッセージへフォールバック。
- 正規化: `feasibility_score` を 0–100 にクランプ、空要素の除去、要約/推奨の既定補完。

## 15. 非機能要件・制約

- 距離・移動時間の計算は地図 API 非依存（座標直接入力・直線距離近似）。実運用では実地図ルーティングへの差し替えを想定し、移動時間算出を `TravelTimeMatrixBuilder` に分離。
- 足順の地図表示は Leaflet ＋ OpenStreetMap タイルを使用するため、地図描画にはインターネット接続が必要。
- 単一日・単一調査員・当日内時刻。
- アップロードサイズ上限 10MB。アップロードファイルはタイムスタンプ名で保存。
- 警告ゼロ運用（AGENTS.md）、フィールドの `_` 接頭辞禁止。
