# route と route2 の処理結果の差異および訪問順の優劣に関する考察

作成日: 2026-06-17

`RoutePlanner` には、足順(訪問順)の決め方が異なる 2 つの実装がある。

- **route**（本プロジェクト）: 前処理として決定的アルゴリズムで順序と数値を確定し、LLM はレビューのみを担う。
- **route2**: 前処理を行わず、訪問順序・各 stop の時刻/距離/移動時間/違反まで全てを LLM が生成する。

本書は両者の結果が一致しない理由と、訪問順としてどちらが優れているかを考察する。

---

## 1. アーキテクチャの違い（事実確認）

| 観点 | **route**（前処理あり） | **route2**（LLM 一括） |
|---|---|---|
| パイプライン | CSV読込 → `RouteOptimizer.Optimize`(アルゴリズム) → `RouteReviewAnalyzer`(LLMはレビューのみ) | CSV読込 → `RoutePlanGenerator.Generate`(LLMが順序も数値も生成) |
| 訪問順を決めるのは | 決定的アルゴリズム | LLM |
| 距離/時刻/移動時間を計算するのは | コード(`TravelTimeMatrixBuilder` ＋ `SimulateOrder`) | LLM |
| LLM の役割 | 出来上がった順序の**採点・改善提案**(順序は変えない) | **順序生成 ＋ 全数値算出 ＋ レビュー**を一括 |
| 実行可能性の保証 | あり(制約を破る順序は棄却) | なし(LLM 任せ) |
| 決定性 | プランは完全決定的 | 構造的に非決定的になりうる |

要点は次の対比に表れている。

- route: `RoutePlannerService.cs` で `optimize → review` の二段（`Optimize` の結果が最終プラン、`review` はそれに対する注釈）。
- route2: `RoutePlannerService.cs` で `generator.GenerateAsync` 一発（順序も数値も LLM 出力）。

---

## 2. 差異が発生する理由

### A. 順序決定ロジックが別物
- **route**（`Services/RouteOptimizer.cs`）: 貪欲最近傍法で初期順序を作り（`BuildInitialOrder`）、**2-opt / Or-opt の局所探索**（`ImproveOrder`）で交差を解消する。採用基準は明示的に `違反数 → 残業 → 総移動時間 → 総距離` の辞書式（`IsBetter`）。さらに地域結束ペナルティ・距離タイブレーク・時間帯遵守ボーナスを持ち、改善が尽きるまで回すので**局所最適に到達**する。
- **route2**: LLM が座標から空間関係を“推論”して順序を出す。プロンプト（`Prompts/route_planner.txt`）に式や「A→C→B の行き来を避けよ」とは書いてあるが、**2-opt 相当の探索を脳内で実行するわけではない**ため交差が残りやすい。

### B. 数値計算の主体が別物（最大の差異源）
- **route**: `TravelTimeMatrixBuilder.Build` がハバーサイン × 道路係数 ÷ 速度 × 天候係数を**厳密計算**し、`SimulateOrder` が分単位で時刻を積算する。`到着 = 前stop出発 + 移動`、`合計 = 各legの和` が**必ず整合**する。
- **route2**: 同じ式を**LLM が暗算**する。90〜120 点規模（バンドルされたサンプル `kofu-dense-90 / window-100 / tight-120`）のハバーサインと分単位積算は、LLM が正確にやれる領域を超えており、**近似と累積ドリフト**が出る。結果として「到着時刻が前 stop の出発+移動と合わない」「leg 合計と total_distance が食い違う」といった内部不整合が起こりうる。

加えて、route2 の `RoutePlanGenerator.BuildPlan` は LLM が返した到着/出発/違反フラグを**そのまま採用**する。件数の数え直しと「total が 0 なら各 leg を合算」というフォールバックだけで、**時刻の再計算も実行可能性の再検証も行わない**。つまり LLM の誤りがそのまま画面に出る。

### C. 制約遵守の強度が別物
- **route**: Strict 枠に間に合わない／稼働終了上限を超える順序は `SimulateOrder` が `null` を返して**棄却**し、入らない訪問は未割当へ。採用される順序は必ず実行可能。
- **route2**: Strict 遵守・残業・昼休憩はすべて LLM の自己申告。違反数は**LLM が立てた `window_violation` フラグを数えるだけ**なので、LLM がフラグを立て忘れれば「実際は遅延しているのに違反 0」になりうる。コードはそれを止めない。

### D. 決定性・再現性
- **route** のプランは LLM が数値に関与しないので**同一入力 → 完全に同一出力**。モデル更新の影響を受けない。
- **route2** は `Temperature=0` でも、モデル/基盤の更新やプロンプトの微差で順序が変わりうる**構造的非決定性**を持ち、監査・回帰テストが難しい。

### E. 規模依存の劣化
- 90〜120 件は組合せ的に巨大。決定的アルゴリズムは余裕で処理できるが、LLM は座標推論と暗算が破綻しやすい領域に入る。件数が増えるほど差が拡大する。

> 補足: `Document/RouteOptimizer-Algorithm.md` は「このアルゴリズムを LLM が再現できるよう命令形に書き起こしたもの」で、route2 はまさにその思想の実装。ただし **“式を渡す” ことと “式を正確に実行する” ことは別**、というのが差異の本質である。

---

## 3. 訪問順としてどちらが優れているか

**中核目的（移動最小化 × 時間枠遵守）においては route が明確に優れる。** 特に今回の 90〜120 件規模では差が顕著。

### route の優位点
- 実行可能性が**構造的に保証**される（Strict 違反・稼働超過は採用されない）
- 2-opt / Or-opt で**交差が除去**され、移動が短く一貫
- **決定的で監査可能**、回帰テストしやすい
- **限界コストほぼゼロ・高速**（トークン消費なし）

### route2 の利点と限界
- 利点: `accessNote / hazardNote / appointmentRequired` のような**自然言語の文脈制約を定性的に勘案できる柔軟性**。小規模なら妥当な順序も出せる。
- 限界: 大規模では交差が残り、数値が破綻し、枠違反・残業を取りこぼし、実行ごとに揺れる。**自分の出した順序を再シミュレーションしない**ため、その順序の良否を自分で保証できない。

### 推奨
本番の訪問順は **route** を採用すべき。理想は**役割分担**であり、これは route が既に体現している — 決定的エンジンが順序と数値を確定・検証し、LLM は「採点・リスク注記・人が読む改善提案」という LLM が得意な仕事だけを担う（`RouteReviewAnalyzer`）。

route2 を活かすなら、現状の最大の欠陥（**検証の欠如**）を埋めるべき。すなわち「LLM が順序を決める → その順序を `SimulateOrder` / `TravelTimeMatrixBuilder` で**決定的に再計算し、実行可能性を再検証**する」ハイブリッドにすれば、LLM の文脈的柔軟性を残しつつ、数値の信頼性と枠遵守の保証を取り戻せる。
